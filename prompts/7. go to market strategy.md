**PROJECT CONTEXT**

**Product:** OpenHuman — a Pixar-style Digital Human streaming system. An open SDK that lets any app embed a realtime animated AI character (facial expressions + voice) via WebSocket. Free tier at scale; targeting 100M+ MAU as the de-facto standard (think: "MP3 for digital humans").

**Core technical architecture:**

- Two separate models: OpenHuman-Motion (text → 135 facial curves) + OpenHuman-Voice (phoneme → mel-spectrogram)
- Custom binary WebSocket protocol with delta motion encoding + anti-crawl layer
- Elixir/OTP + Cowboy backend on Hetzner; Cloudflare edge CDN
- In-process Tokio Rust KV cache (no Redis); >70% cache hit rate target
- Custom render engine (WebGL2 / Vulkan / Metal) — NOT three.js / Unity / Unreal
- Cross-device memory via OpenHuman-Memory + Postgres

**MVP scope (intentionally narrow):**

- Single language pair: Vietnamese + English (first market build)
- Single fixed Pixar character, single voice, single motion style
- No multi-character, no multi-voice at MVP
- Rationale: maximizes cache hit rate, minimizes inference cost, enables free tier

**Strategic MOATs being built:**

1. **Data MOAT** — motion extraction pipeline from YouTube (10,000+ hours of paired audio/transcript/135-curve data) that competitors cannot easily replicate
2. **Format MOAT** — OpenHuman-Protocol becomes the de-facto streaming standard for digital humans, the way MP3 became the audio standard
3. **Cost MOAT** — >70% traffic served from cache; per-user cost ~$0.001/month vs. competitors' ~$0.10/month; free tier is permanently sustainable

**Why Pixar style (not realistic):**

- Avoids uncanny valley; no deepfake/biometric risk; 100× lower bandwidth than realistic 3D streaming; YouTube data is sufficient quality
- Optimal tradeoff: visual presence + free-tier scalability + safety

**Current phase:** Pre-launch / closed beta (1k–10k users). Target public launch in ~4 months (Phase 7 in roadmap).

**Strategic go-to-market model:** Open-source the render engine + SDK (Mapbox model). Embed → network effect → data flywheel → standard lock-in → cost MOAT.
